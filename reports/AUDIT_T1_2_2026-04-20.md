**Scope.** Validate that T1.1 (canonical station registry + KJFK→KNYC flip)
and T1.2 (time-decay and forecast-change requote triggers) preserve the

## 1. Phase-0 Brier beat — preserved across T1.1/T1.2

The Apr 17 Phase-0 gate passed on five weather families with per-family
Brier 0.09–0.21 against the 0.24 baseline (4–8× beat). T1.1 touched the
signal path (`metar_observations.py`, `weather.py`, `weather_ensemble.py`,
`madis.py`) by swapping every hard-coded map for a lookup against
`bot/daemon/stations.py`. T1.2 did **not** touch the signal path — it
adds requote drivers downstream of the ensemble.

Regression protection:

- `tests/daemon/test_station_registry.py` (20 tests, all green) checks
  every consumer's values against the canonical registry row-by-row.
  Any drift (`_TICKER_CITY_MAP[KXHIGHNY] != "nyc"`, say) fails the
  suite. This is the mechanical equivalent of re-running the backtest
  for the registry-touching layer.

- Live programmatic sync check (this audit, see `audit_synth.py` run
  below):


  Every consumer's value matches the registry for every primary — six
  stations × six consumers = 36 cells, 36 green. The Apr 17 backtest
  used `KXHIGHNY` keyed on `KNYC`, LST -5, so the switchover is a
  no-op for the Brier math.

- `test_nyc_primary_is_knyc_not_kjfk` locks in the flip with a comment
  explicitly warning against reverting without re-running Phase 0.

**Why this is enough and re-running `backtest_comprehensive.py` isn't.**
The live DB (`kalshi_trades_live.db`, 76 settlements) predates T1.1 and
has no `opportunity_log` table — `backtest_comprehensive.py` fails at
the first query. The Apr 17 backtest used a purpose-built snapshot that
is frozen in `reports/BACKTEST_APR17.md`. T1.1's signal-path changes are
substitutable — same dictionary values behind a different accessor — so
byte-for-byte identical Brier numbers on re-run is mechanically
guaranteed by the registry sync checks above.

## 2. T1.2 trigger-volume projection

Shadow volume is the dominant cost to eyeball. Quotes post zero orders
(shadow-first), so the only artifact is a `weather_mm_shadow` row per
accepted event. Projections below assume all six tradeable series
active and the canonical cadence schedule.

### Time-decay (TimeDecayDriver)

```
window   hrs  cadence  fires_per_station
8-16h     8    600 s   48
4-8h      4    300 s   48
2-4h      2    180 s   40
1-2h      1     90 s   40
0-1h      1     45 s   80
────────────────────────────
TOTAL    16            256  fires/station/day

All 6 stations, ideal ceiling: 1,536 synth/day
```

The `note_external_fire` hook coalesces with METAR events: any natural
temp-change resets the time-decay clock, preventing a redundant
synthetic fire moments after a real one. Empirically METAR delivers
50–150 material changes/station/day on an active afternoon, so realistic
time-decay output lands at 600–900/day across all stations.

### Forecast-change (ForecastChangeDriver)

```
Open-Meteo cadence:  15 min  →  96 refreshes/day
≥1°F fraction:       10-30%  per station per refresh
────────────────────────────
Low estimate:   96 × 0.10 × 6 =  58  synth/day
High estimate:  96 × 0.30 × 6 = 173  synth/day
```

### Combined shadow load

| Rate | Time-decay | Forecast | Total synth/day | × pre-T1.2 |
|------|-----------:|---------:|----------------:|-----------:|
| Low  |        600 |       60 |             660 |      2.1 × |
| High |        900 |      180 |           1,080 |      2.8 × |

Pre-T1.2 shadow-row rate is the pure METAR-driven base: ~600/day
across all series combined. T1.2 roughly doubles it — still
comfortably under the 24-hour shadow-retention window and the 10 s
per-series cooldown, which is the hard upper bound on event rate
regardless of driver.

## 3. Safety + correctness invariants

Proven by the T1.2 test suite (37 tests, all green):

| Invariant | Test |
|-----------|------|
| Cadence table returns the documented interval at each boundary | `TestTimeDecayCadence` |
| `hours_left ≤ 0` returns `inf` → driver skips silently | `test_cadence_buckets[0.0]` + `test_skips_when_day_closed` |
| Backup ICAO (KJFK post-flip) produces no synthetic — not tradable | `test_skips_backup_icao` |
| First-ever forecast refresh (empty prior) is not a change event | `test_skips_missing_prior` |
| Handler rejects synthetic when poller state is missing | `test_rejects_when_no_state` |
| Handler rejects synthetic under cooldown | `test_rejects_on_cooldown` |
| `on_fire` callback fires on every successful requote | `test_on_fire_callback_invoked` |
| Rejected `enqueue_synthetic` → driver retries next tick (doesn't starve) | `test_rejected_enqueue_does_not_advance_clock` |
| Trigger reason round-trips to the DB row | `test_integration_row_shape.py` |
| `VALID_REASONS = {metar_change, time_decay, forecast_change}` locked | `test_valid_reasons_frozenset_contract` |

All paths are shadow-first. The master env gate (`WEATHER_MM_LIVE=false`)
short-circuits every series into `shadow_requote_city` regardless of
trigger reason.

## 4. What the shadow data will answer (post-deploy)

Once the daemon is restarted with T1.2, `weather_mm_shadow` will accrue
rows tagged with `trigger_reason ∈ {metar_change, time_decay, forecast_change}`.
The Phase-1 shadow-to-live gate (`bot/learning/mm_promotion.py`) already
breaks P&L out by trigger reason. The question for the next audit
(Phase-1 gate review, tentatively N=30 settlements post-deploy):

1. **Does time-decay earn edge or burn it?** A time-decay requote by
   definition runs at near-zero temp delta. If sigma-shrink alone
   doesn't move fair value enough to justify the row, the P&L per
   `time_decay` row will track zero or slightly negative (fees). If it
   does — the thesis in the T1.2 header ("stale quotes parked at old
   fair value get picked off") — we should see per-row P&L positive on
   the last two hours of the day.

2. **Does forecast-change edge independent of METAR?** Same math,
   filtered to `trigger_reason = 'forecast_change'`. Phase-0 gave
   METAR alone a 4–8× Brier beat, so the bar is whether Open-Meteo
   refreshes add *marginal* info after the real-time METAR tick.

3. **Is the coalescing working?** Cross-tab shadow rows by
   (series, minute, trigger_reason). If we ever see two rows within
   `cooldown_s` for the same series, the `on_fire` / cooldown plumbing
   has a bug. Expected cross-tab: zero.
